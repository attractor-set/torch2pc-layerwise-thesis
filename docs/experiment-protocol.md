# Экспериментальный протокол

[English version](experiment-protocol_EN.md)

```text
статические проверки
-> подготовка ресурсов
-> закрепление Torch2PC
-> C0/C1 CPU
-> C0/C1 GPU
-> пилотная матрица только на валидационной выборке
-> фиксация пилотной конфигурации
-> итоговые парные запуски
-> диагностика
-> подготовка отчётов
```

Для парного сравнения совпадают индексы разбиения, `model_seed`, случайное
начальное значение загрузчика, оптимизатор, размер пакета, тип данных, образ,
коммит Torch2PC и файлы набора данных.

Неудачный [запуск](glossary.md#term-run) получает отдельный `run_id`, сохраняет метаданные отказа и не
перезаписывает предыдущий [запуск](glossary.md#term-run). Повтор не удаляет исходную ошибку.

Итоговое [выполнение](glossary.md#term-execution) блокируется при отсутствии файлов контрольных проверок или
манифеста фиксации пилотной конфигурации.

## Повторный доступ к тестовой выборке

Для одной комбинации исходного коммита, разрешённой конфигурации и
`model_seed` разрешается один завершённый итоговый [запуск](glossary.md#term-run). Технически
неудачную попытку можно повторить, но её запись и причина сохраняются. Новый
успешный [запуск](glossary.md#term-run) после изменения кода или конфигурации считается отдельным
экспериментом и не заменяет исходный [подтверждающий анализ](glossary.md#term-confirmatory-analysis).

## Артефакты наблюдения

Предсказания на валидационной и тестовой выборках сохраняются для каждого
примера вместе с исходными индексами, истинными метками, предсказаниями и
вероятностями. Это позволяет проверить агрегированные метрики независимо от
исполняемой контрольной точки.
