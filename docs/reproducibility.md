## Детерминизм

`torch.use_deterministic_algorithms` включается в строгом режиме. Если операция
не поддерживает детерминизм, [запуск](glossary.md#term-run) должен завершиться ошибкой либо получить
отдельную документированную конфигурацию. Предупреждение без остановки не
считается достаточным для основного анализа.

## Повторяемость и воспроизводимость

- повторяемость: тот же узел, образ, коммит и [конфигурация](glossary.md#term-configuration);
- воспроизводимость: другой совместимый узел с теми же артефактами;
- статистическая устойчивость: независимо обученные модели с разными
  значениями `model_seed`.

Совпадение отдельных значений с плавающей точкой между разными GPU не
требуется; сравниваются заранее заданные допуски и статистические выводы.

## Файл фиксации среды

`environment-lock.json` связывает контрольные и экспериментальные запуски с:

- SHA-256 исходного кода и конфигураций;
- неизменяемыми идентификатором образа Docker и `RepoDigest`;
- списком пакетов внутри контейнера;
- полным коммитом Torch2PC;
- сведениями об узле, драйвере и инструментах ROCm.

После изменения исходного кода или конфигурации файл фиксации создаётся заново,
а C0/C1 повторяются. Контрольные результаты содержат SHA-256 файла фиксации,
поэтому артефакт другой среды не принимается автоматически.

## Канонические артефакты релизов и байтовая целостность A1 CSV

Тяжёлые артефакты завершённых этапов 2, 3A и 3B B0 публикуются как отдельные
канонические артефакты в выпусках GitHub. Их отсутствие в автоматически
сформированном архиве исходного кода GitHub не означает отсутствие
доказательств: каждый артефакт выпуска проверяется по опубликованной SHA-256 и
собственному внутреннему манифесту, когда он предусмотрен пакетом.

Наличие этих артефактов выпусков не устраняет отдельную историческую проблему
байтовой воспроизводимости A1 CSV. Манифесты A1 v1 были созданы для шести типов
CSV в каждой из полос `cpu` и `rocm` — всего двенадцати зарегистрированных
путей — когда записи завершались `CRLF`. Git затем нормализовал эти текстовые
файлы и выдаёт их в рабочем дереве и архиве исходного кода с окончаниями строк
`LF`. Поэтому прямой `sha256sum -c` может сообщить о несовпадении только для
этих зарегистрированных CSV, несмотря на неизменность табличного содержимого.

Каноническая проверка выполняется функцией
`verify_a1_evidence_manifest`. Она допускает несовпадение только для точного
закрытого множества из двенадцати A1 CSV и только когда детерминированное
преобразование `LF` в `CRLF` воспроизводит запечатанную контрольную сумму. Для
всех других артефактов требуется точное побайтовое совпадение. Существующие
манифесты A1 и их контрольные суммы не переписываются под нормализованные
`LF`-байты: это изменило бы уже запечатанное доказательство вместо
документирования исторической совместимости.

Таким образом, применяются две независимые политики:

- артефакты выпусков этапов 2, 3A и 3B B0 обеспечивают доступность и проверку
  целостности тяжёлых артефактов соответствующих фаз;
- зарегистрированное правило `CRLF` → Git `LF` обеспечивает ограниченную
  проверяемость старых A1 CSV и не распространяется ни на другие файлы, ни на
  новые пакеты доказательств.

## Проверяемость метрик

Каждый завершённый [запуск](glossary.md#term-run) сохраняет предсказания на валидационной выборке, а
[итоговое выполнение](glossary.md#term-final-execution) — также предсказания на тестовой выборке вместе с исходными
индексами объектов, истинными метками, предсказаниями и вероятностями. Это
позволяет повторно вычислить агрегированные метрики без повторного запуска
модели.

Публичный пакет включает код, конфигурации, библиографию, агрегированные
результаты, рисунки и таблицы, но не PDF публикаций, исходные наборы данных и
тяжёлые контрольные точки.
