# Итоговый протокол этапа 2

[English version](stage-2-protocol_EN.md)

Итоговый этап 2 точно повторяет завершённую подтверждающую матрицу из 80
экспериментальных ячеек с одним контролируемым вмешательством — изменением
реализации Torch2PC.

## Фиксированные факторы

- наборы данных: MNIST и FashionMNIST;
- [архитектура](glossary.md#term-architecture): `lenet_classic`;
- методы: BP, Exact, FixedPred и Strict;
- случайные начальные значения модели: от 0 до 9;
- детерминированные файлы разбиения на обучающую, валидационную и тестовую
  выборки вместе с их SHA-256;
- оптимизатор, скорость обучения, размер пакета, число эпох, тип данных,
  порядок выполнения, контейнерная среда, стек PyTorch/ROCm и целевое
  оборудование;
- FixedPred: `eta=0.1`, `inference_steps=10`;
- Strict: `eta=0.05`, `inference_steps=20`.

## Контролируемое вмешательство

Этап 1 использует исходную редакцию Torch2PC
`00c6c50ee3540537bbb56ab2b6567b541f42b093`. Этап 2 использует отдельно
закреплённый [кандидат](glossary.md#term-candidate), производный от этого коммита. Коммит кандидата
должен быть полным SHA Git и отличаться от редакции этапа 1.

## Изоляция

Этап 2 записывает собственные реестр, каталог запусков, сводки, таблицы, файл
фиксации среды, контрольные проверки допуска, план выполнения и манифест
фиксации. Артефакты этапа 1 используются только как неизменяемые эталонные
[доказательные материалы](glossary.md#term-evidence).

## Обязательные проверки допуска

До запуска матрицы из 80 экспериментальных ячеек должны пройти проверки на CPU
и GPU:

1. исправленный Exact относительно обратного распространения ошибки;
2. исправленный контроль FixedPred относительно исправленного Exact;
3. исходная и исправленная реализации сравниваются по функции потерь,
   производной выхода, состояниям, ошибкам предсказания и градиентам параметров
   для Exact, FixedPred и Strict;
4. рабочие деревья кандидата и эталона соответствуют закреплённым коммитам;
5. образ этапа 2, подготовленные ресурсы, файл фиксации среды, план выполнения и
   манифест фиксации относятся к одним и тем же редакциям исходного кода и
   Torch2PC.

## Основное сравнение

Межверсионный анализ является парным по набору данных, модели, методу и
случайному начальному значению модели. Он публикует абсолютные изменения
времени и памяти, отношения замедления методов относительно BP, различия
качества и контрасты разности различий. Этап 2 не заменяет и не создаёт заново
[доказательные материалы](glossary.md#term-evidence) этапа 1.

## Модель коммитов доказательных материалов

Образ Docker собирается из чистого коммита исходного кода выполнения. Локальный
`prepared_assets.json` создаётся как игнорируемый файл и связывается с файлом
фиксации среды по SHA-256. После создания файла фиксации среды, проверок CPU/GPU
и плана выполнения эти доказательные файлы коммитятся без изменения
закреплённых исходных файлов и конфигураций. Манифест фиксации коммитится
следующим коммитом доказательных материалов. Перед выполнением матрицы проверка
протокола подтверждает, что текущая ветвь является потомком коммита выполнения,
а каждый закреплённый исходный файл и файл конфигурации по-прежнему совпадает с
файлом фиксации среды.

После завершения всех 80 запусков `prepared_assets.json`, реестр этапа 2,
сводки, таблицы и манифесты добавляются в итоговый коммит доказательных
материалов. Такой порядок сохраняет связь образа Docker с коммитом выполнения и
позволяет версионировать созданные [доказательные материалы](glossary.md#term-evidence) без циклической
пересборки файла фиксации среды.

## Закрытие матрицы

После выполнения команда `make snapshot-stage2` проверяет ровно 80 уникальных
успешных экспериментальных ячеек, наличие тестовой оценки для каждой ячейки,
единство редакций исходного кода и Torch2PC, распределение 40/40 по наборам
данных и 20/20/20/20 по методам. Она создаёт неизменяемый снимок реестра,
запись SHA-256 и сводку завершения. Отчёты этапа 2 и межверсионный анализ
строятся из этого снимка.
