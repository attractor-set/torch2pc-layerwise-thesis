# Этап 3: послойный диагностический контур

[English version](stage3-layerwise-analysis_EN.md)

## Назначение

Этот контур добавляет два исследования только на валидационной выборке:

1. **Проверка градиентов при одинаковом состоянии** — BP, Exact, FixedPred и
   Strict получают одну контрольную точку и один и тот же пакет
   валидационной выборки. Шаг оптимизатора не выполняется. Градиенты
   сравниваются отдельно по параметрам и по верхнеуровневым слоям.
2. **Проверка представлений** — независимо обученные контрольные точки
   сравниваются на одном фиксированном наборе примеров валидационной выборки с
   помощью линейного CKA, RSA и полной межслойной матрицы CKA.

Диагностические запуски используют перехватчики и копирование тензоров, поэтому
их измерения времени и памяти не являются эталонными измерениями
производительности. Этапы 1 и 2 остаются неизменными; новые наблюдения относятся
только к этапу 3.

## 1. Проверка установки

```bash
python -m pytest tests/unit/test_layerwise.py
python -m ruff check \
  src/torch2pc_thesis/layerwise.py \
  scripts/run_stage3_same_state_probe.py \
  scripts/run_stage3_representation_probe.py \
  scripts/aggregate_stage3_layerwise.py \
  tests/unit/test_layerwise.py
```

## 2. Поиск контрольной точки

Контрольные точки этапов 1 и 2 не хранятся в Git. Используйте распакованный
набор воспроизведения или локальные артефакты. Для канонической проверки при
одинаковом состоянии используйте парную контрольную точку BP:

```bash
find results -type f -name 'checkpoint.pt' | sort
```

## 3. Проверка градиентов при одинаковом состоянии

```bash
python scripts/run_stage3_same_state_probe.py \
  --checkpoint /absolute/path/to/checkpoint.pt \
  --checkpoint-label final \
  --output results/stage3/layerwise/pilot/seed-0/final \
  --enforce-exact-control
```

Результаты:

- `gradient_metrics.csv` — одна строка на пакет, метод, область и
  слой/параметр;
- `gradient_summary.csv` — среднее по диагностическим пакетам внутри одной
  модели;
- `method_losses.csv` — диагностическая функция потерь методов;
- `metadata.json` — [набор данных](glossary.md#term-dataset), модель, случайное начальное значение, хэш
  индексов валидационной выборки и результат проверки Exact с BP.

Основные поля:

- `cosine` — совпадение направления;
- `relative_l2` — относительная ошибка полного вектора;
- `norm_ratio` — отношение нормы режима предиктивного кодирования к норме BP;
- `sign_agreement` — доля совпавших знаков;
- `max_abs_difference` — максимальное абсолютное расхождение.

Отдельно обученная модель остаётся независимой статистической единицей. Строки
разных слоёв и пакетов не следует трактовать как независимые повторения.

## 4. Проверка представлений

Для корректного сравнения контрольные точки должны иметь одинаковые набор
данных, модель, случайное начальное значение разбиения и долю валидационной
выборки.

```bash
python scripts/run_stage3_representation_probe.py \
  --checkpoint bp=/path/to/bp/checkpoint.pt \
  --checkpoint exact=/path/to/exact/checkpoint.pt \
  --checkpoint fixedpred=/path/to/fixedpred/checkpoint.pt \
  --checkpoint strict=/path/to/strict/checkpoint.pt \
  --output results/stage3/layerwise/pilot/seed-0/final
```

Результаты:

- `representation_metrics.csv` — CKA и RSA для соответствующих слоёв;
- `cross_layer_cka.csv` — полная матрица «слой BP × слой кандидата»;
- `activations_<label>.npz` — матрицы активаций, согласованные по примерам;
- `metadata.json` — список контрольных точек, слоёв и хэш индексов
  валидационной выборки.

Слои `0`, `1`, `3`, `4`, `5` соответствуют верхнеуровневым блокам текущего
`lenet_classic`. При изменении модели список необходимо зафиксировать в
отдельной конфигурации этапа 3 до подтверждающего запуска.

## 5. Агрегирование нескольких моделей

Организуйте артефакты по случайному начальному значению и контрольной точке,
затем выполните:

```bash
python scripts/aggregate_stage3_layerwise.py \
  --root results/stage3/layerwise/pilot \
  --output results/stage3/layerwise/pilot/combined
```

Скрипт объединяет таблицы, но намеренно не выполняет статистическую проверку.
Сначала агрегируйте диагностические пакеты внутри каждой модели, затем
проводите парный анализ по отдельно обученным моделям.

## 6. Рекомендуемое пилотное исследование

- [набор данных](glossary.md#term-dataset): FashionMNIST;
- модель: `lenet_classic`;
- независимо обученные модели: 3;
- пакеты проверки при одинаковом состоянии: 5;
- примеры проверки представлений: 1000;
- [контрольная точка](glossary.md#term-checkpoint): лучшая итоговая [контрольная точка](glossary.md#term-checkpoint);
- загрузчик тестовой выборки: не создаётся.

После проверки объёма данных, устойчивости контроля Exact и отсутствия
вырожденного CKA зафиксируйте:

- слои;
- контрольные точки;
- число пакетов и примеров;
- семейства гипотез;
- пороги численной эквивалентности;
- правила исключения некорректных наблюдений.

## 7. Ограничение первой версии

Исправление анализирует доступные контрольные точки и итоговые градиенты
режимов предиктивного кодирования. Оно не восстанавливает отсутствующие
промежуточные контрольные точки этапов 1 и 2 и не извлекает состояния после
каждой внутренней итерации вывода Torch2PC.

Для анализа траектории создайте отдельную обучающую кампанию этапа 3 с
сохранением контрольных точек на заранее заданных долях обновлений
оптимизатора. Трассировки каждого шага вывода следует добавлять отдельным
инструментирующим исправлением Torch2PC после фиксации схемы наблюдений, чтобы
не смешивать корректность диагностики с изменением пути выполнения
предиктивного кодирования.
