# Этап 3B: интегрированная модель фронтира

[English version](stage3b-integrated-frontier-model_EN.md)

**Статус:** проектная фиксация после `ADR-039`; [выполнение](glossary.md#term-execution), создание `oracle`-меток, сбор признаков и управление остаются закрытыми.

## 1. Назначение

Документ вводит единую модель состояния для исследования достаточности
`FixedPred`, стоимости наблюдения и будущей оркестрации `QWake-PC`. Модель не
меняет математическую динамику `FixedPred`, точный эталон `stage2_baseline` или
`EX-IF0`. Она только разделяет:

1. где находится каноническое вычисление;
2. какой объём `pre-action` информации уже приобретён;
3. какое действие предлагается относительно текущего фронтира.

`ADR-039` остаётся исторически и нормативно неизменным. Его алфавит
`DONE / UNKNOWN / SWEEP` сохраняется как семантика теневого допуска и
неопределённости. Новый алфавит ниже является более точным алфавитом действий.

## 2. Разделение архитектурных уровней

Роли не смешиваются:

```text
PC-TREF  = семантика достаточности, допустимого regret и допуска действия
PC-CATM  = механизмная модель и источник интерпретируемых признаков
QWake-PC = оркестрация зарегистрированных переходов фронтира
executor = каноническая траектория stage2_baseline
```

`PC-TREF` отвечает на вопрос, допустимо ли принять текущий фронтир относительно
зарегистрированного конечного ответа. `PC-CATM` может предоставить признаки
механизма коррекции, но не разрешает действие. `QWake-PC` может только выбрать
между заранее зарегистрированными переходами и до отдельного допуска работает
с `controls_execution=false`.

Отрицательный результат конкретной оркестрации не опровергает `PC-TREF` или
`PC-CATM`.

## 3. Интегрированный фронтир

Для обязательного временного пути состояние фронтира обозначается

```math
\mathcal F_{t,i}=
\left(S_t, R_{A_i}, B^{\mathrm{compute}}_{\mathrm{rem}},
B^{\mathrm{diag}}_{\mathrm{rem}}, P\right),
```

где:

- $S_t$ — неизменяемый снимок после $t$ канонических свипов;
- $R_{A_i}$ — накопительное `pre-action` представление уровня наблюдения;
- $B^{\mathrm{compute}}_{\mathrm{rem}}$ — остаток вычислительного бюджета;
- $B^{\mathrm{diag}}_{\mathrm{rem}}$ — остаток диагностического бюджета;
- $P$ — происхождение, идентичность модели, данных, `RNG`, графа и реализации.

В будущем пространственном расширении к идентичности фронтира может быть
добавлен заранее зарегистрированный вложенный агрегат $B_k$. Это расширение не
входит в обязательное временное ядро и не открывает выполнение рекурсивных
агрегатов.

Фронтир является монотонным: переход не может отменить выполненный свип,
удалить приобретённый уровень или изменить происхождение снимка.

## 4. Ось наблюдения `A0 / A1 / A2` и уровень `O`

### `A0` — структурное наблюдение

`A0` содержит только заранее доступные структурные поля: идентификаторы
снимка, метода, свипа, слоя и блока, зарегистрированную топологию, размеры,
`dtype`, бюджет и членство в заранее заданных агрегатах. Чтение значений
тензоров не требуется.

### `A1` — дешёвые редукции на устройстве

`A1` добавляет конечный зарегистрированный набор редукций над уже доступными
тензорами. Время устройства, синхронизации, переданные байты и временная память
измеряются, а не считаются нулевыми.

### `A2` — локальные редукции на детерминированной подвыборке

`A2` добавляет локальные редукции по индексам, выведенным только из
`pre-action` идентичности и замороженного ключа. Подвыборки образуют вложенное
семейство

```math
S_0\subseteq S_1\subseteq\cdots\subseteq S_q.
```

Ключ, размеры и порядок фиксируются до создания меток.

### `O` — только `post-action oracle`

`O` создаётся полным каноническим суффиксом на отдельно замороженной
подвыборке:

```math
U_O\subseteq U_{A2}\subseteq U_{A1}=U_{A0}.
```

`O` не является осью доступного фронтира, не входит в развёртываемое
представление и никогда не передаётся теневому контроллеру.

Накопительные представления фиксированы:

```text
R_A0 = A0
R_A1 = A0 + A1
R_A2 = A0 + A1 + A2
O    = post-action labels only
```

## 5. Алфавит действий

Интегрированная модель использует ровно три верхнеуровневых действия.

### `ACCEPT_FRONTIER`

Принять текущий вычислительный фронтир как достаточный для
зарегистрированного конечного ответа. Действие допустимо только после
положительного `PC-TREF`-допуска по безопасности и риску. Стоимость, `NCZ`,
`ECZ`, малый остаток или компонентное завершение сами по себе не разрешают
`ACCEPT_FRONTIER`.

### `ADVANCE_FRONTIER`

Выполнить ровно один зарегистрированный смежный переход:

1. приобрести следующий уровень наблюдения при неизменном $S_t$; или
2. выполнить один следующий канонический свип и открыть новый фронтир
   $S_{t+1}$ с базовым уровнем `A0`.

Тип перехода, целевой идентификатор и полный [вектор стоимости](glossary.md#term-cost-vector) сохраняются
явно. Переход не может пропускать незарегистрированный уровень, выполнять
несколько свипов или выбирать цель по `post-action` метке.

### `COMPLETE_SUFFIX`

Выполнить полный оставшийся канонический суффикс `stage2_baseline` до
зарегистрированного эталонного горизонта. Это единственный `fail-closed` резервный
путь при исчерпании допуска, диагностического бюджета, реестра переходов или
целостности состояния.

В текущем пакете все три действия являются только теневыми предложениями.
Фактическая траектория остаётся полным каноническим суффиксом.

## 6. Отображение `D / U / S`

`ADR-039` не переписывается. Используется следующее совместимое отображение:

| `ADR-039` | Интегрированная интерпретация |
|---|---|
| `DONE` | [кандидат](glossary.md#term-candidate) `ACCEPT_FRONTIER`, требующий положительного допуска |
| `UNKNOWN` | эпистемическое состояние; само по себе не является действием |
| `UNKNOWN` с допустимым следующим уровнем | `ADVANCE_FRONTIER(observation)` |
| `SWEEP` как один следующий канонический свип | `ADVANCE_FRONTIER(compute)` |
| неразрешённая неопределённость или нарушение контракта | `COMPLETE_SUFFIX` |

Таким образом, `UNKNOWN` больше не перегружает действие приобретения, а
`SWEEP` не обозначает одновременно один свип и полный резервный суффикс.

## 7. Граф переходов обязательного ядра

Для каждого $t$ заранее регистрируется граф:

```text
F(t, A0) --acquire_A1--> F(t, A1)
F(t, A1) --acquire_A2--> F(t, A2)
F(t, Ai) --one_sweep----> F(t+1, A0)
F(t, Ai) --accept-------> terminal_shadow_accept
F(t, Ai) --full_suffix--> terminal_canonical_reference
```

Отсутствующий, недоступный или слишком дорогой смежный переход не заменяется
скрытой альтернативой. В таком случае действует `COMPLETE_SUFFIX`.

Первая обязательная теневая каскадная схема остаётся детерминированной:

```text
A0 -> A1 -> A2 -> ADVANCE_FRONTIER(compute) | COMPLETE_SUFFIX
```

`Greedy` порядок, обучаемый выбор аналитики, активная пространственная
локальность и спайкоподобная оркестрация не входят в обязательное ядро.

## 8. Учёт стоимости переходов

Стоимость относится к ребру фронтира, а не только к конечному решению:

```math
\Delta\mathbf C(e)=
\left(
C_{\mathrm{host}}, C_{\mathrm{device}}, N_{\mathrm{sync}},
B_{\mathrm{d2h}}, M_{\mathrm{temp}}, B_{\mathrm{trace}}
\right).
```

Обязательные поля:

```text
host_elapsed_ns
device_elapsed_ns
synchronization_events
device_to_host_bytes
temporary_peak_bytes
persistent_trace_bytes
acquisition_count
```

Сохраняются маржинальная и накопительная стоимости `A0`, `A1`, `A2`, одного
канонического свипа и полного суффикса. Стоимость создания `O` учитывается
отдельно как исследовательская стоимость разметки и не включается в
развёртываемую диагностическую стоимость.

Экономия рассматривается только после допуска безопасности:

```text
safety -> safe coverage -> UNKNOWN burden -> full cost vector -> stability
```

## 9. Невмешательство и граница времени

Приобретение наблюдения обязано быть только читающим и не изменять состояние,
параметры, буферы, градиенты, `RNG`, `autograd`-путь или выбор канонической
ветви. Любая мутация является нарушением архитектуры.

```text
observer_state_mutations=0
observer_parameter_mutations=0
observer_buffer_mutations=0
observer_rng_mutations=0
observer_autograd_path_changes=0
observer_interference_events=0
post_action_feature_leakage=0
```

До действия доступны только поля текущего фронтира. $M^*$, $t^*$, конечный
`endpoint`, фактическая полезность следующего свипа и `oracle`-оптимальный
переход являются `post-action` информацией.

## 10. Граница утверждений

Эта фиксация разрешает только документационное проектирование и последующую
синтетическую проверку чистых типов. Она не разрешает научный сбор, создание
меток, `A11-OFF0`, активацию политики, [доступ к тестовой выборке](glossary.md#term-test-dataset-access) или утверждение
об ускорении.
