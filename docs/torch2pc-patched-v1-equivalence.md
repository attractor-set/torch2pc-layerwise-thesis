# Torch2PC `patched-v1`: изменения реализации и обоснование математической эквивалентности

[English version](torch2pc-patched-v1-equivalence_EN.md)

## 1. Назначение документа

Документ описывает изменения, внесённые в реализацию Torch2PC для заключительного этапа 2, и обосновывает, почему эти изменения:

1. сохраняют математические определения исследуемых методов;
2. сохраняют вычисляемые состояния и градиенты в зарегистрированной области эксперимента;
3. применимы к архитектуре, данным и протоколу проекта `torch2pc-layerwise-thesis`;
4. изменяют главным образом вычислительный путь, количество повторных проходов графа автоматического дифференцирования и связанные накладные расходы.

Документ не утверждает универсальную формальную эквивалентность для всех возможных модулей PyTorch. Вывод ограничен закреплёнными версиями исходного кода, контролируемой средой и классом моделей, проверенным в эксперименте.

## 3. Математические инварианты

Набор изменений сохраняет определения и порядок вычислений, существенные для исследуемых методов.

### 3.1. Модель прямого прохода и функция потерь

Набор изменений не меняет:

- состав и порядок слоёв `torch.nn.Sequential`;
- функции слоёв;
- параметры модели;
- функцию потерь;
- входные и целевые тензоры;
- оптимизатор и правила обновления параметров в проекте диссертации.

Следовательно, прямое отображение модели и оптимизируемая функция остаются прежними.

### 3.2. Exact

Для Exact сохраняется связь между активацией прямого прохода, ошибкой и состоянием:

```text
belief = activation - epsilon
```

Изменение объединяет повторные вычисления в один прямой и один согласованный обратный проход. Вычисляется производная той же функции по тем же активациям и тензорам параметров. Меняется организация вызовов автоматического дифференцирования, а не само дифференцируемое выражение.

### 3.3. FixedPred

Для FixedPred сохраняются выражения:

```text
epsilon = fixed_activation - belief
state update = epsilon - VJP(next_error)
```

Точка линеаризации остаётся фиксированной на протяжении вывода состояний. Набор изменений один раз строит локальные графы VJP в этой точке и повторно применяет их на шагах вывода состояний. Такое преобразование устраняет повторное построение эквивалентных графов и сохраняет якобиан, к которому применяется векторно-якобианово произведение.

### 3.4. Strict

Для Strict сохраняются выражения:

```text
epsilon = local_prediction - belief
state update = epsilon - VJP(next_error)
```

Ошибки повторно вычисляются на каждом шаге вывода состояний, а обновление выполняется в исходном обратном порядке. Набор изменений повторно использует выход уже выполненного локального прямого прохода только внутри одной итерации и не переносит предсказание между разными шагами вывода состояний.

### 3.5. Градиенты параметров

До внесения изменений градиенты отдельных параметров одного модуля могли вычисляться несколькими вызовами `torch.autograd.grad`. После изменения параметры модуля передаются в один вызов.

Для параметров \(\theta_1,\ldots,\theta_k\) преобразование меняет:

```text
grad(y, theta_1), ..., grad(y, theta_k)
```

на:

```text
grad(y, (theta_1, ..., theta_k))
```

Оба варианта вычисляют компоненты одной производной \(\partial y / \partial \theta\). Объединённый вызов сокращает число обратных проходов, не меняя функцию, переменные дифференцирования или входной вектор для VJP.

## 4. Подробное описание изменений

### 4.1. Корректность программного интерфейса и повторное использование входов

Внесены следующие изменения:

- пользовательское значение `vinit` передаётся в Strict;
- готовые активации и значение функции потерь Exact корректно используются повторно;
- проверки `== None` заменены на `is None`.

В конфигурациях экспериментов этапов 1 и 2 используется `vinit=None`. Поэтому исправление обработки явно заданного `vinit` расширяет корректность программного интерфейса, но не меняет исследуемый путь выполнения.

### 4.2. Совместное вычисление VJP параметров

Внесены следующие изменения:

- градиенты всех обучаемых параметров одного последовательного модуля вычисляются совместно;
- модули без параметров пропускаются;
- локальный граф освобождается после завершения требуемого VJP.

Сохраняются:

- тензоры, относительно которых вычисляется производная;
- скалярный или векторный выход;
- аргумент `grad_outputs`;
- соответствие каждого градиента конкретному параметру.

### 4.3. Exact с одним прямым проходом

Внесены следующие изменения:

- используется один прямой проход;
- ошибки активаций и градиенты параметров извлекаются согласованным обратным проходом;
- устраняются повторный прямой проход и последовательность отдельных обратных проходов.

Сохраняются значения активаций, функция потерь, дифференцируемый граф и итоговые производные.

### 4.4. Повторное использование VJP в FixedPred

Внесены следующие изменения:

- локальный граф создаётся один раз в фиксированной точке линеаризации;
- обратное отображение повторно применяется в течение `n` шагов вывода состояний;
- граф освобождается после последнего использования;
- `detach()` применяется там, где требуется отделить историю вычислений, а независимая копия хранилища не создаётся, если она не нужна математике метода.

Сохраняются фиксированное предсказание, якобиан в точке линеаризации и последовательность обновлений состояний.

### 4.5. Повторное использование локального прямого прохода в Strict

Внесены следующие изменения:

- выход одного локального прямого прохода используется и для ошибки предсказания, и для VJP той же итерации;
- устраняются избыточные вызовы `model.zero_grad()` внутри вывода состояний;
- сокращаются операции клонирования и копирования;
- сохраняются повторный прямой проход на каждом шаге вывода состояний и исходный обратный порядок обновления.

`torch.autograd.grad` возвращает градиенты без накопления в полях `.grad` параметров. Поэтому удаление `zero_grad()` из внутреннего пути не меняет вычисляемый VJP.

### 4.6. Проверки и микроизмерение производительности

Для кандидата добавлены:

- проверки эквивалентности исходной и исправленной реализаций;
- проверка Exact исправленной реализации относительно BP;
- регрессионная проверка явно заданного `vinit`;
- микроизмерение производительности, совместимое с CPU и GPU.

Локальный набор проверок кандидата:

```text
5 passed
```

Микроизмерение используется как диагностический инструмент. Научные выводы о времени выполнения строятся по полной парной матрице этапов 1 и 2.

## 5. Структурное доказательство сохранения алгоритма

Автоматизированная структурная проверка закреплённого исходного кода кандидата подтвердила, что:

- Strict пересчитывает ошибки на каждой итерации;
- Strict использует выражение `prediction - belief`;
- обновление Strict имеет форму `epsilon - VJP`;
- FixedPred использует выражение `activation - belief`;
- обновление FixedPred имеет форму `epsilon - VJP`;
- Exact использует выражение `activation - epsilon`.

Все структурные наблюдения имеют значение `true` в контрольных артефактах CPU и GPU:

```text
results/stage-2/summaries/control_gate_cpu.json
results/stage-2/summaries/control_gate_gpu.json
```

Структурная проверка показывает, что ключевые формулы и порядок алгоритма присутствуют в исходном коде кандидата. Она дополняется численным сравнением состояний и градиентов.

## 7. Достаточность доказательств для данного эксперимента

Этап 2 использует:

- `lenet_classic`;
- MNIST и FashionMNIST;
- методы BP, Exact, FixedPred и Strict;
- случайные начальные значения `0..9`;
- те же разбиения наборов данных;
- те же гиперпараметры методов;
- тот же конвейер обучения;
- закреплённый образ PyTorch/ROCm;
- AMD Radeon RX 7700 XT;
- коммиты исходной и исправленной реализаций, закреплённые полными SHA.

Контрольные проверки выполняются на том же типе последовательной архитектуры и в том же программно-аппаратном стеке, что и заключительная матрица. Они сравнивают не только конечную скалярную метрику, но и промежуточные состояния и градиенты параметров. Поэтому проверки непосредственно охватывают механизм, влияющий на обучение.

Для интерпретации этапа 2 применяется следующий вывод:

> В закреплённой экспериментальной области `patched-v1` сохраняет вычисляемые состояния и градиенты исходного Torch2PC в пределах предварительно зарегистрированных численных допусков. Следовательно, `patched-v1` может использоваться как [кандидат](glossary.md#term-candidate), сохраняющий математическое поведение реализации, для оценки влияния устранения вычислительных накладных расходов.

## 8. Изменения, ожидаемые после исправления реализации

Ожидаются изменения следующих инженерных показателей:

- [астрономическое время](glossary.md#term-wall-time) обучения;
- среднее и медианное время эпохи;
- количество проходов автоматического дифференцирования;
- пиковый объём выделенной и зарезервированной памяти GPU;
- соотношение времени выполнения BP, Exact, FixedPred и Strict.

Эти изменения составляют предмет этапа 2 и не рассматриваются как нарушение математической эквивалентности.

Небольшие различия конечных метрик обучения между независимыми запусками возможны из-за вычислений float32, ядер GPU и накопления ошибок округления. Поэтому качество оценивается по полной парной матрице, а не по отдельной ячейке.

## 9. Ограничения вывода

Полученные доказательства не распространяются автоматически на:

- произвольные модели вне проверенного последовательного пути выполнения;
- стохастические слои, работающие в других режимах;
- пользовательские функции автоматического дифференцирования с побочными эффектами;
- модули, зависящие от изменения тензоров на месте или совместного использования их хранилища;
- градиенты высших порядков;
- другие версии PyTorch, ROCm или CUDA;
- другие коммиты Torch2PC;
- режимы явно заданного `vinit`, кроме отдельно добавленной регрессионной проверки;
- гиперпараметры и процедуры вывода состояний, не представленные в протоколе контрольных проверок.

Для расширения области применимости необходимо повторить структурные и численные контрольные проверки в новом закреплённом окружении.

Ключевые математические определения Exact, FixedPred и Strict, порядок обновления состояний, дифференцируемые функции и производные параметров сохраняются. Это подтверждается:

1. анализом изменений;
